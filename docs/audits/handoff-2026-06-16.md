### Known gap — storage objects are never deleted (minor, not a blocker)

`storage.objects` has no DELETE policy, and the app's image-remove / submission-
reject paths only delete the catalog row (`bromide_images`) or flip submission
status — they never call `storage.remove()`. So removed/rejected image files stay
in the `bromides` bucket and remain reachable by direct URL. Impact: storage
bloat over time, and rejected (potentially unwanted) images stay public. Fix =
add an admin DELETE policy on the bromides bucket + call `storage.remove()` on
image-remove and reject. No user flow is broken by this.

## Decision A — RESOLVED: warn admin on destructive settings change (objective #4)

Chosen approach: warn-on-destructive. `CollectionEditor.save` now computes how
many already-registered images would be detached by the new slot layout
(`orphanedImageCount`, unit-tested in `orphanedImageCount.test.ts`) and, when
nonzero, shows a confirm dialog ("登録済み画像が枠から外れます … 続行しますか？")
before committing. Additive edits save silently as before. Verified in a real
browser: uploading an image then switching the collection from member_grid to
flat surfaces "1枚の登録済み画像が … 切り離されます"
(`dogfood-output/upload-confirm-verify/screenshots/destructive-edit-warning.png`).

A full image-entity refactor (A2) remains the longer-term ideal if you want
edits to never detach anything; the warning is the agreed interim.

### Original analysis

`buildStableSlots` (`src/components/admin/CollectionEditor.tsx:99`) matches
previous slots only by the freshly derived `legacyId`. Unmatched previous slots
are silently dropped. So additive edits (add member, add size, raise count)
preserve images/ownership, but destructive edits orphan them:

- renaming a size label (`L` → `L版`)
- switching `kind` (`member_grid` ↔ `flat` ↔ `mixed`)
- reassigning an item's member/no/type

`buildBromides` renders one tile per `slots` entry, so naively keeping the old
slots would produce phantom tiles. A correct fix is a design choice:

- (A1) keep unmatched slots as hidden/retired entries that still resolve images
  but do not render — least disruptive, moderate code.
- (A2) full image-entity model: images are independent rows with tags, the
  collection is a projection — matches the stated intent, large refactor.
- (A3) accept additive-only and warn the admin before a destructive change.

Not changed unilaterally because all three alter data semantics. Needs a call.
