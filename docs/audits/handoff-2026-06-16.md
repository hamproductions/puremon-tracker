## Decision A — destructive collection-settings change still orphans data (objective #4)

`buildStableSlots` (`src/components/admin/CollectionEditor.tsx:99`) matches
previous slots only by the freshly derived `legacyId`. Unmatched previous slots
are silently dropped. So additive edits (add member, add size, raise count)
preserve images/ownership, but destructive edits orphan them:

- renaming a size label (`L` → `L版`)
- switching `kind` (`member_grid` ↔ `flat` ↔ `mixed`)
- reassigning an item's member/no/type

`buildBromides` renders one tile per `slots` entry, so naively keeping the old
slots would produce phantom tiles. A correct fix is a design choice:

- (A1) keep unmatched slots as hidden/retired entries that still resolve images
  but do not render — least disruptive, moderate code.
- (A2) full image-entity model: images are independent rows with tags, the
  collection is a projection — matches the stated intent, large refactor.
- (A3) accept additive-only and warn the admin before a destructive change.

Not changed unilaterally because all three alter data semantics. Needs a call.

## Decision B — live Supabase DB is not migrated (objective #6 + security)

`bun run verify:supabase-public` against the live project (test_user) shows:

```
anonymous profiles read blocked: FAIL  (anyone can read all profiles)
non-admin profile admin escalation blocked: FAIL  (any user can self-promote to admin)
selected-oshi insert/readback/delete: FAIL  (public.oshi missing)
preference upsert/readback/delete: FAIL  (public.user_preferences missing)
```

The first two are live security holes. All four are fixed by applying two
idempotent SQL files — they only add tables/columns and tighten policies:

- `supabase/live-persistence-hotfix-2026-06-16.sql` → `public.oshi`,
  `public.user_preferences`, `collections.slots`
- `supabase/live-policy-hotfix-2026-06-16.sql` → profile RLS + admin-escalation
  guard/trigger

This cannot be done from here: no Supabase CLI, no DB password, no service-role
key — only the anon key (which cannot run DDL). Apply via the Supabase SQL
editor, then re-run:

```
SUPABASE_TEST_EMAIL=test_user@ham-san.net SUPABASE_TEST_PASSWORD='<pw>' bun run verify:supabase-public
```

After that, the cross-session / cross-user ownership+oshi+preference persistence
matrix can be run for real against the live DB.
